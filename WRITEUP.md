# WRITEUP

## Architecture & approach

The core decision is a **two-tier trust model**. Numbers come from the SEC XBRL
CompanyFacts API — filer-tagged, machine-readable, never touched by the LLM.
Narrative comes from filing HTML, retrieved and cited. This is what lets me make
a strong claim: *the dashboard cannot hallucinate a financial figure*, because no
figure is ever generated by a language model. The LLM phrases pre-computed
numbers and reasons over retrieved text, nothing more.

Files: `sec_client.py` (EDGAR/XBRL ingestion + provenance), `metrics.py` (derived
metrics with formula + source traceability and conflict detection), `rag.py`
(chunking, local embeddings, Chroma retrieval), `agent.py` (LangGraph routing →
tool → self-check → refuse), `app.py` (Streamlit), `eval/` (labeled set + runner).
Stack is fully local via Ollama (`qwen2.5:7b-instruct`, `nomic-embed-text`): $0,
no keys, reproducible.

## Hallucination rate — would I trust it in front of an executive?

For the **numeric** path, yes — with a caveat I can defend. Numeric answers are
deterministic reads of filer-tagged XBRL with the accession number shown, so the
hallucination rate there is structurally ~0; my build-time checks were 6/6 exact
match against hand-verified ground truth and 3/3 correct on comparative leader.
The risk is not invented numbers but **wrong-concept selection** (e.g. picking a
segment figure when the user meant consolidated) — which is why every answer
exposes the exact XBRL tag and accession for one-glance verification.

For the **narrative** path I am more cautious, and an executive should be too: a
7B local model can still mis-summarize even when retrieval is correct. That is
why narrative answers run a grounding self-check and refuse on failure. The honest
hallucination number for that path comes from `eval/run_eval.py` on the
unanswerable set — run it and paste the result; I did not fabricate it.

**What I'd fix first:** retrieval precision. Section-aware chunking helps, but a
reranker over the top-k and a hybrid keyword+vector retriever would cut the main
remaining narrative failure mode (plausible-but-off-topic passages) more than any
prompt change.

## A failure I found and diagnosed

Three real ones, all caught by testing against live data:

1. **Quarterly/annual collision.** My first metrics index keyed facts on
   `(concept, fy)`. XBRL tags prior-year *quarterly* values with the *current*
   filing's `fy`, so a single 10-K dumped ~11 revenue entries under one `fy`. This
   inflated "restatement conflicts" to 162 for NVDA. **Fix:** select annual flows
   by period *duration* (~365 days) and balance-sheet items by period-end date,
   keyed by the year the period actually ends in. Conflicts dropped to 0 false /
   a handful of genuine ones.

2. **Tag-switch revenue gap.** NVDA reports revenue under
   `RevenueFromContractWithCustomerExcludingAssessedTax` in older filings and
   `Revenues` recently. My initial code picked the first alias that appeared
   *anywhere* and stopped — so revenue silently vanished for FY2023+ and the
   latest year showed only debt-to-equity. **Fix:** collect from all aliases and
   resolve per period by priority. NVDA revenue is now continuous and correct
   (FY2024 \$60.9B → FY2025 \$130.5B → FY2026 \$215.9B, each verified against the
   actual filings).

3. **Silent wrong-year fallback.** "NVDA gross margin in 1850" returned FY2026
   because the classifier ignored the year and defaulted to latest — a quiet
   wrong answer, the worst kind. **Fix:** parse the requested fiscal year; if it
   isn't available, refuse and list the years that are.

## How I used AI tooling — and where I overrode it

I used an AI coding assistant throughout to draft modules and scaffold the
LangGraph graph. The clearest **override**: the assistant initially proposed
parsing financial figures out of the filing **text/tables with the LLM** and using
those as the numbers behind the charts. I rejected that outright — it is the exact
pattern that produces confident wrong figures — and rebuilt the quantitative spine
on the XBRL CompanyFacts API so numbers are filer-tagged, not model-extracted.
A second override: an initial eval grader I accepted extracted "the first number"
from an answer and was scoring the *fiscal year* ("FY2024") instead of the value;
I caught it because correct answers were failing, and rewrote extraction to prefer
unit-tagged values and skip calendar-year tokens.

## Where the framework helped, and where I dropped down

**LangGraph helped** with the agent's control flow — explicit nodes for
classify/rewrite/route/self-check made the refusal logic auditable and let me
expose a reasoning trace in the UI for free. The state machine is the right shape
for "route to a tool, then verify."

**Where I dropped to raw API calls:** the LLM and embedding calls go **straight to
the Ollama REST API** rather than through a LangChain chat wrapper. The wrappers
added indirection and version-coupling I didn't want for two simple endpoints
(`/api/chat`, `/api/embeddings`), and raw calls made the local JSON-mode
self-check and temperature-0 determinism trivial to control. Likewise the XBRL
layer is plain `requests` against `data.sec.gov` — no framework belongs between me
and the authoritative numeric source, where I want full control over caching,
rate-limiting, and exactly which tag/period I trust.
